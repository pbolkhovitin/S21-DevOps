**Перевод масок:**

**Ручной расчет и преобразование масок**

#### 1.1.1 Маска `255.255.255.0` → Префиксная (`/24`) и Двоичная запись

**Расчёт:**

- **Обычная запись:** `255.255.255.0`

- **Перевод в двоичную систему:**
  * `255` (десятичное) = `11111111` (двоичное, 8 бит)
  * `0` (десятичное) = `00000000` (двоичное, 8 бит)

- **Двоичная маска:** `11111111.11111111.11111111.00000000`

- **Подсчёт единичных бит (префикс):**
  * Первые три октета (байта) по 8 единиц: `8 + 8 + 8 = 24`
  * Четвёртый октет: `0` единиц
  * **Итого префикс:** `/24`

**Результат:**
  * Обычная: `255.255.255.0`
  * Префиксная: `/24`
  * Двоичная: `11111111.11111111.11111111.00000000`

#### 1.1.2 Маска `/15` → Обычная (`255.254.0.0`) и Двоичная запись

**Расчёт:**

- **Префиксная запись:** `/15`

- **Построение двоичной маски:**
  * Первые `15` бит — единицы, остальные `17` бит — нули.

  - **Распределение по октетам (по 8 бит в каждом):**
    * Октет 1: `8` единиц → `11111111` (десятичное: `255`)
    * Октет 2: `7` единиц (`15 - 8 = 7`) → `11111110` (десятичное: `254`)
    * Октет 3: `0` единиц → `00000000` (десятичное: `0`)
    * Октет 4: `0` единиц → `00000000` (десятичное: `0`)

- **Двоичная маска:** `11111111.11111110.00000000.00000000`

- **Перевод в десятичную (обычную) запись:**
  * `11111111` (бинарный) = `255` (десятичный)
  * `11111110` (бинарный) = `254` (десятичный)
  * `00000000` (бинарный) = `0` (десятичный)
  * `00000000` (бинарный) = `0` (десятичный)
  * **Итого обычная запись:** `255.254.0.0`

**Результат:**
  * **Префиксная:** `/15`
  * **Обычная:** `255.254.0.0`
  * **Двоичная:** `11111111.11111110.00000000.00000000`

#### 1.1.3 Маска `11111111.11111111.11111111.11110000` → Обычная (`255.255.255.240`) и Префиксная (`/28`)

**Расчёт:**

- **Двоичная запись:** `11111111.11111111.11111111.11110000`

- **Перевод в десятичную (обычную) запись по октетам:**
    * Октет 1: `11111111` = `255`
    * Октет 2: `11111111` = `255`
    * Октет 3: `11111111` = `255`
    * Октет 4: `11110000` = `240`
      + Позиции битов в октете (слева направо): `128`, `64`, `32`, `16`, `8`, `4`, `2`, `1`
      + `11110000` = `128 + 64 + 32 + 16 = 240`
    * **Итого обычная запись:** `255.255.255.240`

**Результат:**
- **Двоичная:** `11111111.11111111.11111111.11110000`
- **Обычная:** `255.255.255.240`
- **Префиксная:** `/28`

## Part 3. Утилита iperf3

В данной части работы выполнялось измерение скорости передачи данных между виртуальными машинами **ws1** и **ws2** с помощью утилиты `iperf3`. Предварительно были выполнены расчёты по переводу единиц измерения скорости передачи данных.

### 3.1. Скорость соединения — перевод единиц измерения

Для корректного понимания результатов работы сетевых утилит необходимо уметь переводить значения скорости между различными единицами измерения. 

**Выполненные расчёты:**

- **8 Mbps в MB/s:**
   - 1 байт = 8 бит
   - 8 мегабит в секунду (Mbps) = 8 / 8 = **1 мегабайт в секунду (MB/s)**

- **100 MB/s в Kbps:**
   - 1 MB/s = 1024 KB/s (по стандарту IEC, либо 1000 KB/s по десятичной системе; в сетевых расчётах часто используется 1 MB = 1000 KB для упрощения)
   - 100 MB/s = 100 × 1000 = **100 000 KB/s**
   - 1 KB/s = 8 Kbps
   - 100 000 KB/s = 100 000 × 8 = **800 000 Kbps**

- **1 Gbps в Mbps:**
   - 1 гигабит в секунду (Gbps) = **1000 мегабит в секунду (Mbps)**

### 3.2. Измерение скорости соединения с помощью iperf3

Для измерения пропускной способности канала между машинами использовалась утилита `iperf3`. Перед началом работы была проведена проверка наличия утилиты на обеих машинах и, при необходимости, выполнена её установка:

```bash
# Установка iperf3, если не установлено
```

Принцип работы `traceroute`:

Утилита `traceroute` использует поле TTL (Time To Live) в IP-пакетах для обнаружения промежуточных маршрутизаторов. Алгоритм работы:

Отправляет пакет с TTL=1. Первый маршрутизатор уменьшает TTL до 0, отбрасывает пакет и отправляет обратно ICMP-сообщение Time Exceeded (тип 11, код 0). Адрес отправителя этого сообщения — IP маршрутизатора.

Затем отправляет пакет с TTL=2 — второй маршрутизатор возвращает ICMP Time Exceeded, и так далее.

Когда пакет достигает целевого хоста с TTL=1 (или больше), хост возвращает ICMP Port Unreachable (так как `traceroute` использует заведомо недоступный порт UDP) либо Echo Reply (для ICMP-режима).

На дампе `tcpdump` видна последовательность:

Пакеты с увеличивающимся TTL от **ws11** (`10.10.0.2`)

Ответы ICMP Time Exceeded от **r1** (`10.10.0.1`) — первый хоп

Ответы ICMP Time Exceeded от **r2** (`10.100.0.12`) — второй хоп

Итоговый ответ от **ws21** (`10.20.0.10`) — третий хоп

На дампе tcpdump видны запросы Echo (ICMP type 8) от **ws11** и ответы Destination Unreachable от **r1**.

**Вывод:** При отсутствии маршрута к целевой сети маршрутизатор возвращает отправителю ICMP-сообщение об ошибке, и пакет не передаётся дальше.

#### 7.2.3. Добавление правил SNAT и DNAT (правила 5-6)
В финальной версии файла были добавлены правила для **SNAT** и **DNAT** :

**SNAT** (Source Network Address Translation) — преобразование исходного адреса пакетов, выходящих из локальной сети `10.20.0.0/26` во внешнюю сеть. Пакеты маскируются под адрес внешнего интерфейса роутера **r2** .

**DNAT** (Destination Network Address Translation) — преобразование адреса назначения для входящих соединений на порт `8080` роутера **r2**, которые перенаправляются на веб-сервер **ws22** (порт `80`) .

```bash
#!/bin/sh
iptables -F
iptables -X
iptables -t nat -F
iptables -t nat -X

iptables --policy FORWARD DROP
iptables --policy INPUT ACCEPT
iptables --policy OUTPUT ACCEPT

# 1. Разрешить ICMP (ping)
iptables -A FORWARD -p icmp -j ACCEPT

# 2. Разрешить установленные соединения
iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT

# 3. SNAT - маскирование локальной сети
iptables -t nat -A POSTROUTING -s 10.20.0.0/26 -o ens18 -j MASQUERADE

# 4. DNAT - доступ к ws22 извне
iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.20:80

# 5. Разрешить новые соединения К ws22 (для DNAT)
iptables -A FORWARD -p tcp -d 10.20.0.20 --dport 80 -j ACCEPT

# 6. Разрешить новые соединения ОТ ws22 (для SNAT)
iptables -A FORWARD -s 10.20.0.0/26 -p tcp --dport 80 -j ACCEPT

# 7. Разрешить ответный трафик
iptables -A FORWARD -d 10.20.0.0/26 -m state --state ESTABLISHED -j ACCEPT
```

Пояснение правил:

    SNAT (MASQUERADE): iptables -t nat -A POSTROUTING -s 10.20.0.0/26 -o ens18 -j MASQUERADE — все пакеты из сети 10.20.0.0/26, выходящие через интерфейс ens18, получают исходный адрес этого интерфейса (адрес r2 в сети 10.100.0.0/16). MASQUERADE удобен тем, что автоматически работает при изменении IP-адреса внешнего интерфейса.
    
    DNAT: iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.20:80 — все входящие TCP-пакеты на порт 8080 роутера r2 перенаправляются на веб-сервер ws22 (10.20.0.20), порт 80.
    
    FORWARD для ESTABLISHED,RELATED: iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT — разрешает пропуск обратных пакетов для уже установленных соединений.
    
    FORWARD для порта 80 ws22: iptables -A FORWARD -p tcp -d 10.20.0.20 --dport 80 -j ACCEPT — разрешает форвардинг новых соединений, направленных на веб-сервер ws22.
    
    FORWARD для исходящих соединений: iptables -A FORWARD -s 10.20.0.0/26 -p tcp --dport 80 -j ACCEPT — разрешает форвардинг новых исходящих соединений из сети 10.20.0.0/26 к веб-серверам (например, для проверки SNAT).

    - Stateful inspection — использование состояний соединений (`ESTABLISHED,RELATED`) для обеспечения обратной связи по уже установленным соединениям, что критически важно для корректной работы NAT.
    
    - Итеративный подход — настройка firewall выполнялась поэтапно с проверкой каждого добавленного правила, что является правильной практикой при конфигурировании сетевых экранов.
    
    Настроенные правила NAT обеспечивают полноценную маршрутизацию и доступ к внутренним ресурсам извне, что является ключевым навыком при администрировании сетей.

## Part 8. Дополнительно. Знакомство с SSH Tunnels

В данной дополнительной части работы изучалась технология SSH-туннелирования для организации доступа к веб-серверу, запущенному на локальном интерфейсе **ws22**, с других машин сети. SSH-туннели позволяют безопасно перенаправлять трафик TCP через зашифрованное SSH-соединение.

### 8.1. Подготовка инфраструктуры

#### 8.1.1. Запуск фаервола на r2

В соответствии с заданием, на роутере **r2** был запущен фаервол с правилами из Part 7, которые включают настройки SNAT и DNAT для обеспечения маршрутизации между сетями.

**Обновление файла `/etc/firewall.sh` на машине r2 дополеннными правилами**
```bash
#!/bin/sh

# 1. Очистка всех цепочек в таблицах filter и nat
iptables -F
iptables -F -t nat
iptables -X
iptables -X -t nat

# 2. Политика по умолчанию: запрещаем FORWARD (транзитные пакеты)
iptables -P FORWARD DROP
# Разрешаем входящие и исходящие на самом роутере (чтобы ssh и т.д. работали)
iptables -P INPUT ACCEPT
iptables -P OUTPUT ACCEPT

# 3. Разрешаем транзитную маршрутизацию для протокола ICMP (пинги)
iptables -A FORWARD -p icmp --icmp-type echo-request -j ACCEPT
iptables -A FORWARD -p icmp --icmp-type echo-reply -j ACCEPT

# ВАЖНО: Добавляем правило, чтобы разрешить форвард уже установленных соединений
# Это нужно для корректной работы TCP-трафика (в т.ч. SNAT/DNAT)
iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT

# 4. Настройка NAT (таблица nat)
# SNAT: Маскируем все пакеты из сети 10.20.0.0/26, уходящие через интерфейс eth1 (в сторону r1)
# Предполагаем, что интерфейс на r2, смотрящий в сторону r1, называется eth1 (нужно проверить командой ip a)
iptables -t nat -A POSTROUTING -s 10.20.0.0/26 -o eth1 -j MASQUERADE

# DNAT: Проброс порта. Все на 8080 порт r2 перенаправляем на ws22:80
iptables -t nat -A PREROUTING -i eth1 -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.20:80

# 5. Разрешаем новые соединения для FORWARD (чтобы DNAT/SNAT могли создавать новые сессии)
# Это правило должно идти после правил ICMP, но до финального запрета (которого у нас нет)
iptables -A FORWARD -m state --state NEW -j ACCEPT
```

Пояснение правил `firewall` на **r2** для Part 8:
    Очистка правил: iptables -F и iptables -t nat -F — удаление всех существующих правил в таблицах filter и nat для создания чистой конфигурации.
    
    Политика FORWARD: iptables -P FORWARD DROP — запрещает маршрутизацию всех транзитных пакетов по умолчанию (требуется явное разрешение).
    
    Разрешение ICMP: iptables -A FORWARD -p icmp --icmp-type echo-request -j ACCEPT и iptables -A FORWARD -p icmp --icmp-type echo-reply -j ACCEPT — разрешают прохождение ICMP-запросов и ответов через роутер (для диагностики).
    
    Разрешение установленных соединений: iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT — пропускает обратные пакеты для уже установленных соединений (критически важно для работы TCP).
    
    SNAT (MASQUERADE): iptables -t nat -A POSTROUTING -s 10.20.0.0/26 -o ens18 -j MASQUERADE — все пакеты из сети 10.20.0.0/26, выходящие через интерфейс ens18, получают исходный адрес этого интерфейса.
    
    DNAT: iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.20:80 — перенаправляет входящие TCP-пакеты на порт 8080 роутера r2 на веб-сервер ws22 (10.20.0.20:80).
    
    Разрешение новых соединений: iptables -A FORWARD -m state --state NEW -j ACCEPT — разрешает создание новых соединений через роутер (необходимо для SNAT/DNAT).
    
    Разрешение SSH на самом r2: iptables -A INPUT -p tcp --dport 22 -j ACCEPT — открывает доступ к SSH-серверу на самом роутере.
    
    Разрешение SSH через r2: iptables -A FORWARD -p tcp --dport 22 -j ACCEPT и iptables -A FORWARD -p tcp --sport 22 -j ACCEPT — разрешают прохождение SSH-трафика через роутер (для туннелей).

### 8.2. Local TCP forwarding (с ws21 до ws22)
`Local TCP forwarding` (локальная переадресация портов) позволяет перенаправлять трафик с локального порта клиента через SSH-сервер на целевой хост и порт .

#### 8.2.1. Создание SSH-туннеля с ws21 до ws22
На машине **ws21** была выполнена команда для создания локального SSH-туннеля до **ws22** через роутер **r2**:

```bash
# На ws21
ssh -f -N -L 8080:localhost:80 wynafryn@10.20.0.20
```

Разбор команды:

    -f — запуск в фоновом режиме после аутентификации

    -N — не выполнять удалённые команды (только проброс портов)
    
    -L 8080:10.20.0.20:80 — опция локальной переадресации портов:
    
    8080 — локальный порт на ws21, который будет слушать SSH-клиент
    
    10.20.0.20 — целевой сервер ws22 (Apache)
    
    80 — порт на целевой машине (ws22), куда перенаправляются подключения
    
    wynafryn@10.20.0.1 — SSH-сервер (роутер r2), через который устанавливается туннель
   
    Что происходит:
    
    SSH-клиент на ws21 подключается к роутеру r2 (10.20.0.1)
    
    Создаётся локальный порт 8080 на ws21, слушающий входящие соединения
    
    Когда кто-то подключается к 127.0.0.1:8080 на ws21, трафик перенаправляется через r2 на ws22:80

### 8.3. Remote TCP forwarding (c ws11 до ws22)
`Remote TCP forwarding` (удалённая переадресация портов, также известная как `reverse forwarding`) позволяет перенаправлять трафик с порта удалённого SSH-сервера на локальный клиент или другой хост в его сети .

#### 8.3.1. Настройка SSH-сервера для удалённого форвардинга
Для работы удалённой переадресации портов необходимо, чтобы на SSH-сервере (в данном случае на **ws22**) была разрешена TCP-переадресация. По умолчанию в OpenSSH сервере параметр `AllowTcpForwarding` установлен в `yes` .

#### 8.3.2. Создание удалённого SSH-туннеля с ws11 до ws22
На машине **ws22** была выполнена команда для создания удалённого SSH-туннеля до **ws11** через роутер **r1** и **r2**:

```bash
# На ws22
ssh -f -N -R 8080:localhost:80 wynafryn@10.10.0.50
```

Разбор команды:

    -f — запуск в фоновом режиме после аутентификации (терминал можно использовать дальше)
    
    -N — не выполнять удалённые команды (только проброс портов, без интерактивной сессии)
    
    -R 8080:localhost:80 — опция удалённой переадресации портов (remote forwarding):
    
    8080 — порт на удалённой машине (SSH-сервере), который будет открыт для прослушивания
    
    localhost — адрес, на который перенаправляются подключения с точки зрения SSH-клиента (машины, откуда запущена команда)
    
    80 — порт на локальной машине (клиенте), куда будут перенаправляться подключения
    
    wynafryn@10.10.0.50 — SSH-сервер, к которому подключается клиент (в данном случае ws11 в сети 10.10.0.0/18)
